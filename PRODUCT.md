# CharShift 产品设计文档（PRD）

> **文档类型**：Product Requirements Document  
> **版本**：v1.0  
> **日期**：2026-05-30  
> **产品名称**：CharShift  
> **定位**：桌面端文本编码批量检测与转换工具

---

## 1. 产品概述

### 1.1 产品背景

在软件开发、系统运维、数据迁移和内容本地化过程中，文本文件的编码不一致是高频痛点。历史项目常使用 GBK、BIG5、ISO-8859-1 等区域编码，而现代工具链默认要求 UTF-8。当需要处理成百上千个文件时，传统方式（命令行工具、编辑器逐个另存）效率极低且容易出错。

**CharShift** 应运而生，它是一款基于 Tauri v2 的本地桌面应用，专为批量处理文本文件编码而设计。用户只需选择目录或直接拖入文件，即可在数秒内完成编码自动检测与批量转换，全程无需上传至网络，确保数据隐私安全。

### 1.2 产品定位

| 维度 | 定位 |
|------|------|
| **用户群体** | 开发者、运维工程师、数据迁移人员、内容编辑者 |
| **核心场景** | 遗留项目编码迁移、日志文件编码统一、多语言文本预处理 |
| **平台策略** | Windows 为主力平台，兼容 Linux 与 macOS |
| **价值主张** | 「一键扫描、自动识别、安全转换」——让批量编码处理从半小时缩短到一分钟 |
| **差异化** | 纯本地运行、自动批量检测、原子写入不损坏原文件、保留文件修改时间 |

### 1.3 产品目标

1. **效率目标**：1000 个文本文件的编码检测与转换总耗时控制在 30 秒以内（普通 SSD 环境）。
2. **安全目标**：转换过程零数据丢失；转换失败时原文件 100% 完好无损。
3. **易用目标**：用户从打开程序到完成首次转换，学习成本为零，无需阅读文档。
4. **稳定目标**：支持单次扫描最多 50,000 个节点不崩溃；正确处理无权限目录和占用中文件。

## 2. 目标用户与用户画像

### 2.1 核心用户群体

#### 用户画像 A：后端开发者「张工」

- **背景**：维护一个十年历史的 Java 企业项目，源代码和配置文件混用 GBK、GB2312、UTF-8。
- **痛点**：每次在新环境构建项目，IDE 打开文件都出现乱码；需要逐个文件调整编码另存，耗时且容易遗漏。
- **需求**：能够递归扫描整个项目目录，自动识别每个文件的原始编码，然后一键全部转为 UTF-8。

#### 用户画像 B：运维工程师「李运维」

- **背景**：管理多台 Linux/Windows 服务器，各服务日志输出编码不统一（有的 GBK，有的 UTF-8，有的 Latin-1）。
- **痛点**：使用 ELK 或脚本分析日志时，因编码不一致导致切割失败或乱码。
- **需求**：快速将某个日志目录下所有文本文件统一为 UTF-8，且保留原有修改时间以便按时间排序分析。

#### 用户画像 C：本地化专员「王翻译」

- **背景**：负责游戏/软件的多语言文本包管理，经常收到外包团队发来的各种编码的 `.txt`、`.csv`、`.po` 文件。
- **痛点**：不知道文件具体是什么编码，用记事本打开试来试去；手动另存为容易改错。
- **需求**：拖入文件即可自动检测编码，确认无误后批量转换为目标编码，界面直观显示每个文件的编码状态。

### 2.2 用户使用场景汇总

| 场景 | 触发方式 | 用户预期 |
|------|----------|----------|
| 项目整体迁移 | 「打开目录」选择项目根目录 | 递归扫描所有子文件夹，识别源编码，批量转换 |
| 零散文件处理 | 从文件管理器拖入文件 | 只处理拖入的文件，自动检测并转换 |
| 转换前确认 | 查看文件树中的编码标签 | 看到每个文本文件被识别出的具体编码，决定是否全选或部分选 |
| 排除干扰项 | 开启「排除二进制文件」 | 自动过滤掉图片、可执行文件等非文本内容，列表更干净 |
| 防止外部修改 | 开启「锁定文件」 | 扫描后文件被锁定，确保转换前不会被其他程序篡改 |

### 3.2 功能需求详述（FR）

#### FR-001 目录选择与递归扫描

- **描述**：用户点击「打开目录」按钮，系统调用原生目录选择对话框；用户选定目录后，后端递归遍历该目录及其子目录。
- **规则**：
  - 最大递归深度 20 层。
  - 自动跳过隐藏文件/目录（以 `.` 开头）和系统目录（如 `System Volume Information`、`$Recycle.Bin`）。
  - 自动跳过符号链接（防止循环遍历）。
  - 最多支持 50,000 个节点，超出时提示用户选择子目录。
- **输出**：文件树结构加载到界面，文本文件与二进制文件以不同视觉样式区分。

#### FR-002 拖放文件添加

- **描述**：用户可将文件或文件批量从系统文件管理器拖入程序主窗口，程序自动识别其中的文本文件并添加到列表。
- **规则**：
  - 拖入的文件中，非文本文件自动过滤并提示。
  - 已存在于列表中的文件自动去重。
  - 拖入的文本文件默认处于「已选中」状态，方便立即转换。
  - 支持 Tauri 原生拖放事件（主要）与 HTML5 拖放（降级兼容）。

#### FR-003 文件树浏览与选择

- **描述**：以树形结构展示目录层级，用户可展开/折叠目录，通过复选框选择需要转换的文件。
- **规则**：
  - 目录节点显示复选框，状态分为：全选、部分选、未选、无子文件。
  - 点击目录复选框可一键全选/取消该目录下所有文本文件。
  - 支持 `Shift + 单击` 进行范围批量选择（以渲染顺序为基准）。
  - 每个节点右侧显示「移除」按钮，可删除该节点及其子节点。
  - 二进制文件复选框置灰，不可选择。

#### FR-004 编码自动检测

- **描述**：对列表中的每个文本文件自动检测其原始编码，并在文件树中显示。
- **检测策略**（优先级递减）：
  1. **BOM 检测**：检查文件头是否有 UTF-8/UTF-16LE/UTF-16BE/UTF-32 BOM。
  2. **频率分析**：读取前 8KB，通过 chardet 进行字符频率统计；置信度 ≥ 0.6 时采纳结果。
  3. **UTF-8 启发式**：若频率分析置信度不足，尝试按 UTF-8 解码，成功则标记为 UTF-8。
  4. **未知编码**：以上均失败，标记为未知。
- **体验要求**：检测过程通过流式进度实时更新界面（每 50 个文件刷新一次状态文字），避免用户感知卡顿。

#### FR-005 目标编码选择与转换

- **描述**：用户从下拉框选择目标编码，点击「开始转换」后，系统对选中的文本文件执行批量编码转换。
- **支持的目标编码**：UTF-8、UTF-8-BOM、UTF-16LE、UTF-16BE、GBK、GB18030、BIG5、ISO-8859-1、WINDOWS-1252。
- **规则**：
  - 仅转换被选中的文本文件；二进制文件和未知编码文件不可被选中转换。
  - 最大并发转换数为 4，防止磁盘 IO 竞争。
  - 转换完成后弹窗告知成功数与失败数；失败文件在列表中显示具体错误原因。

#### FR-006 安全转换机制

- **描述**：转换过程必须保证原文件安全，且转换结果正确。
- **子机制**：
  - **一致性校验**：转换前比对文件大小与修改时间，若与扫描时不一致（说明外部程序已修改），则跳过并提示。
  - **原子写入**：不直接覆盖原文件；先写入带随机后缀的临时文件，写入成功后通过文件系统 `rename` 替换原文件。
  - **元数据保留**：转换成功后，使用 `filetime` 恢复文件的原始修改时间，避免批量转换破坏文件时间线。
  - **错误隔离**：单个文件转换失败不影响其他文件的转换进程。

#### FR-007 文件独占锁定（Windows）

- **描述**：用户可在设置中开启「锁定列表中的文件」。开启后，添加到列表的文本文件将被以独占模式锁定，阻止其他进程在转换前修改。
- **规则**：
  - Windows 下有效；非 Windows 平台开启后仅作为兼容提示，不实际阻止外部修改。
  - 移除节点、清空列表、程序退出时自动解锁对应文件。
  - 锁定失败的文件在状态栏提示数量，不影响其他功能。

#### FR-008 设置管理

- **描述**：提供设置面板，用户可配置两个开关选项。
- **选项**：
  - **扫描时排除非文本文件**：开启后，目录扫描自动跳过二进制文件，列表中仅保留文本文件和目录。
  - **锁定列表中的文件**：见 FR-007。
- **持久化**：设置项使用 `localStorage` 保存，下次启动自动恢复。

#### FR-009 状态反馈与异常提示

- **描述**：界面提供全局状态指示，所有关键操作均有视觉反馈。
- **状态种类**：
  - 就绪（绿色点）
  - 扫描中 / 检测中 / 转换中（橙色点，呼吸动画）
  - 错误（红色点）
- **对话框**：扫描失败、转换失败、无有效文件等场景弹出模态对话框，明确告知原因。

### 4.2 界面布局规范

- **视觉风格**：浅色主题，扁平化设计，圆角卡片（16px）与药丸形按钮（999px）。
- **主色调**：`#FF8400`（橙色），用于品牌标识、主按钮、选中状态高亮。
- **成功色**：`#004D1A`（深绿）；**错误色**：`#8C1C00`（深红）。
- **字体栈**：系统默认无衬线字体，优先 `Segoe UI`、`PingFang SC`、`Microsoft YaHei`。
- **间距体系**：8px 基准网格，按钮高度 40px，工具栏高度 72px，状态栏高度 37px。

### 4.3 关键交互流程

#### 流程 A：目录扫描与转换（标准路径）

```
[启动程序] → 显示空状态
    ↓
[点击「打开目录」] → 弹出系统目录选择框
    ↓
[选择目录] → 按钮禁用，状态变为「正在扫描目录…」
    ↓

    ↓

    ↓

    ↓
[用户勾选/取消文件] → 复选框状态联动（目录级联、Shift 范围选）
    ↓
[选择目标编码] → 下拉框展开，点击选项后收起
    ↓
[点击「开始转换」] → 按钮禁用，状态变为「正在转换 (N 个文件)…」
    ↓

    ↓

```

#### 流程 B：拖放快速处理

```
[从文件管理器选中多个文件] → 拖入程序窗口
    ↓
[触发拖放遮罩] → 窗口中央显示橙色虚线拖放区
    ↓
[释放鼠标] → 遮罩消失，状态变为「正在检测拖入的文件…」
    ↓
[筛选完成] → 文本文件自动添加到根节点并默认选中，非文本文件弹窗提示已跳过
    ↓
[自动检测编码] → 同流程 A 的检测阶段
    ↓
[用户点击「开始转换」] → 同流程 A 的转换阶段
```

#### 流程 C：设置调整

```
[点击标题栏齿轮图标] → 遮罩层 + 设置对话框滑入
    ↓
[切换开关] → 即时生效，自动保存到 localStorage
    ↓
[点击「确定」/ 点击遮罩层] → 对话框关闭
```

## 5. 非功能需求

### 5.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 扫描速度 | 10,000 个文件/秒 | 普通 SSD，不含网络路径 |
| 编码检测速度 | 500 个文件/秒 | 基于 8KB 采样，16 并发 |
| 转换速度 | 200 个文件/秒 | 平均 50KB/文件，4 并发 |
| 界面响应 | ≤ 16ms 卡顿 | 渲染帧率保持 60fps，进度推送每 50 个聚合一次 |
| 内存占用 | ≤ 300MB | 50,000 节点场景下峰值 |

### 5.2 兼容性与平台需求

| 平台 | 最低版本 | 说明 |
|------|----------|------|
| Windows | Windows 10 | 主力平台，独占锁定功能完整可用 |
| macOS | macOS 11 (Big Sur) | 文件锁定为兼容占位 |
| Linux | Ubuntu 20.04 / 主流发行版 | 需 WebKitGTK 运行时 |

### 5.3 稳定性与可靠性

- **原子性**：转换操作必须满足原子性——临时文件成功写入且 `rename` 成功后，才视为转换成功；任何中间失败不得残留临时文件或损坏原文件。
- **容错性**：扫描时遇到无权限目录自动跳过，不中断整体扫描；转换时单个文件失败不影响同批次其他文件。
- **可恢复性**：程序异常退出后，临时文件可能残留（概率极低），但不影响原文件；文件锁定句柄随进程退出自动释放。

### 5.4 安全与隐私

- **纯本地处理**：所有文件内容读取、编码检测、转换写入均在用户本地完成，无网络传输，无云端接口。
- **无数据收集**：程序不收集文件路径、文件名、文件内容或用户行为数据。
- **路径安全**：不接受来自剪贴板、URL 或远程调用的路径输入；所有操作路径仅来源于系统原生对话框或用户拖放。

### 5.5 可访问性

- 界面文字与背景对比度符合 WCAG 2.1 AA 标准。
- 所有交互元素支持键盘焦点（虽然当前版本以鼠标操作为主）。
- 状态指示不仅依赖颜色（成功/错误弹窗带有明确文字说明）。

## 6. 竞品分析与差异化

### 6.1 竞品对照

| 竞品/方案 | 类型 | 批量检测 | 批量转换 | 自动识别编码 | 隐私安全 | 易用性 |
|-----------|------|----------|----------|--------------|----------|--------|
| iconv / recode | 命令行 | ❌ 需手动指定源编码 | ✅ | ❌ | ✅ | ❌ 高学习成本 |
| Notepad++ | 编辑器 | ❌ 逐个文件 | ❌ 逐个另存 | ⚠️ 需手动选择 | ✅ | ⚠️ 中等 |
| VS Code | 编辑器 | ❌ | ⚠️ 可批量但操作复杂 | ⚠️ 自动猜测但不精确 | ✅ | ⚠️ 中等 |
| 在线转换网站 | Web | ❌ 通常单文件 | ❌ 通常单文件 | ⚠️ | ❌ 需上传文件 | ✅ |
| PowerShell 脚本 | 脚本 | ✅ 需编写 | ✅ 需编写 | ❌ | ✅ | ❌ 高门槛 |

### 6.2 CharShift 差异化优势

1. **零配置批量处理**：无需编写脚本，无需记忆命令行参数，开箱即用。
2. **三级编码检测引擎**：BOM + chardet 频率分析 + UTF-8 启发式，识别准确率远高于编辑器的简单猜测。
3. **安全原子写入**：转换过程不直接触碰原文件，失败零副作用；自动恢复修改时间，不破坏版本控制或日志分析的时间线。
4. **Windows 独占锁定**：防止转换前文件被 IDE、编译器或其他进程篡改，降低一致性错误。
5. **原生桌面体验**：基于 Tauri 构建，包体积小、启动快、资源占用低，界面与操作系统原生视觉融合。

## 8. 术语表

| 术语 | 解释 |
|------|------|
| **BOM** | Byte Order Mark，字节顺序标记，位于文本文件开头，用于标识 UTF 编码类型 |
| **chardet** | 基于字符频率统计的编码猜测算法，通过分析文本中各字节的出现概率推断最可能的编码 |
| **原子写入** | 文件写入的原子操作方式：先写入临时文件，再通过文件系统的 rename 操作瞬间替换原文件，确保不会出现半写状态 |
| **独占锁定** | Windows 文件系统中以 `FILE_SHARE_NONE` 模式打开文件，阻止其他任何进程对该文件进行读取或写入 |
| **Tauri IPC** | Tauri 框架提供的前端（WebView）与后端（Rust）进程间通信机制 |
| **流式检测** | 不等待全部文件检测完成再一次性返回结果，而是每检测完一个文件就立即向前端推送进度，提升用户感知速度 |
| **编码转换** | 将文件内容按原始编码解码为 Unicode 字符串，再按目标编码重新编码为字节序列并写入文件的过程 |
